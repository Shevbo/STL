# Sprint02 Phase 2 — план приёмки заявок и maker-исполнения (Slice 1)

Документ для приёмки ордерной части QUIK-агента на боевом счёте.
Источник требований: quik_agent/PHASE2.md. Контракт: proto/shectory/quik/v1/quik_agent.proto.
Провод к QUIK: QLua sendTransaction через localhost TCP-мост.
Стиль и формат повторяют docs/sprint02-acceptance.md.

ВНИМАНИЕ. Это боевой счёт и реальные деньги. Минимальный размер.
Каждое размещение подтверждается оператором в UI. Авто-торговли нет.

## 1. Объём Phase 2 Slice 1

Slice делится на два этапа. Сначала 1a, затем 1b поверх него.

В объёме:
- 1a. Лимитная заявка: выставление, снятие, статус. Один контракт.
  Сообщения PlaceOrder, CancelOrder, OrderUpdate, TransReply.
- 1b. Maker-исполнение цели в N контрактов у своей стороны стакана.
  Сообщения StartExecution, StopExecution, ExecutionUpdate.
- Жёсткие лимиты в агенте и в STL (защита в глубину).
- Прайс-коллар, белый список, дневной лимит, kill-switch.
- Диалог подтверждения оператора в UI "Заявки".

Вне объёма (отложено):
- Рыночные заявки (market). Только лимитные (TYPE=L).
- Перемещение и замена заявки (replace/move) как отдельная команда.
- Стоп-заявки и любые условные заявки.
- Авто-торговля, стратегии, сигналы. Только заявка, решённая человеком.
- Мульти-аккаунт. Один боевой счёт.
- Боевой поток SMS. В Phase 2 только факт интеграции шлюза, не боевой поток.
- Taker-режим. allow_cross по умолчанию false, в Slice 1 не включаем.

Граница Guard 3: агент не выставляет и не снимает заявки без явной
подтверждённой команды STL. KillSwitch снимает всё и блокирует новые.

## 2. Предусловия безопасности

Проверяются ДО первой боевой заявки. Без них приёмка не начинается.

### 2.1 Жёсткие лимиты (агент и STL, одинаковые значения)

- quik_trading_enabled = false по умолчанию. Мастер-флаг. При off любая
  заявка отклоняется в агенте и в STL. Включается только на время приёмки.
- max_contracts_per_order = 2. Больше — REJECTED до QUIK.
- max_working_contracts = 2. Суммарный остаток в стакане.
- price_collar_frac = 0.002. Максимум 0.2% неблагоприятного отклонения
  от цены заявки или цены прихода.
- instrument_whitelist = ["RIU6"]. Любой другой код — REJECTED.
- daily_order_cap. Лимит размещений в день, например 50. Превышение — REJECTED.
- Лимиты живут в коде. Счёт и секреты через keymaster, не в коде.

### 2.2 Kill-switch и подтверждение

- KillSwitch (STL -> агент): снимает все рабочие заявки и блокирует новые
  до явной очистки. Кнопка в UI "Заявки".
- Диалог подтверждения оператора перед каждым размещением. Показывает:
  инструмент, сторону, цену, количество, нотионал и маржу, оценку maker-комиссии.
- Заявка уходит в агент только после подтверждения в диалоге.

## 3. Пошаговая приёмка

Прогон вживую: агент на Windows-боксе, живой QUIK, боевой счёт, RIU6, размер 1-2.
Эталон комиссии: maker (reference_commission_model). Taker в чистом maker-режиме недопустим.

### Этап 1a. Одиночная ручная заявка (1 контракт)

#### Шаг a. Дальний лимит, который не исполнится -> ACTIVE

- Действие: выставить BUY 1 RIU6 лимитом сильно ниже бида (или SELL сильно выше аска).
  Подтвердить в диалоге. quik_trading_enabled = true.
- Ожидание: заявка проходит лимиты, уходит в QUIK, встаёт в стакан.
- Проверка: UI — OrderState PENDING затем ACTIVE. QUIK — заявка видна в "Заявках",
  остаток равен 1. /api — OrderUpdate state=ORDER_STATE_ACTIVE, filled=0, есть order_id.
  TransReply result_code успешный.
- [ ] PASS  [ ] FAIL

#### Шаг b. Снятие -> CANCELLED

- Действие: снять заявку из шага a кнопкой в UI (CancelOrder).
- Ожидание: заявка снята в QUIK, новых остатков нет.
- Проверка: UI — OrderState CANCELLED. QUIK — заявка ушла из активных.
  /api — OrderUpdate state=ORDER_STATE_CANCELLED. max_working_contracts освободился.
- [ ] PASS  [ ] FAIL

#### Шаг c. Лимит у касания -> FILLED, P&L и maker-комиссия

- Действие: выставить лимит у своей стороны касания (BUY по биду, SELL по аску),
  чтобы он исполнился. Размер 1. Подтвердить.
- Ожидание: заявка исполняется как maker. Не пересекает противоположное касание.
- Проверка: UI — OrderState FILLED, filled=1. Показаны P&L и maker-комиссия.
  QUIK — сделка в "Сделках", цена совпадает. /api — OrderUpdate state=ORDER_STATE_FILLED.
  Комиссия maker, сверка по разделу 4.
- [ ] PASS  [ ] FAIL

#### Шаг d. Отклонение биржей (плохая цена/шаг) -> REJECTED с текстом

- Действие: выставить лимит с ценой не по шагу цены (или заведомо невалидной).
- Ожидание: лимиты агента пропускают, но QUIK отклоняет транзакцию.
- Проверка: UI — OrderState REJECTED, виден текст причины. QUIK — OnTransReply
  с кодом ошибки. /api — OrderUpdate state=ORDER_STATE_REJECTED, text непустой;
  TransReply result_code ошибочный, text от QUIK.
- [ ] PASS  [ ] FAIL

#### Шаг e. Пробой коллара -> отклонено агентом ДО QUIK

- Действие: выставить лимит с отклонением от цены прихода больше price_collar_frac.
- Ожидание: агент режет заявку до отправки в Lua/QUIK.
- Проверка: UI — REJECTED, причина "collar". QUIK — транзакции НЕ было,
  в "Заявках" пусто, OnTransReply не приходил. /api — OrderUpdate
  state=ORDER_STATE_REJECTED, text про коллар. В логе моста нет команды place.
- [ ] PASS  [ ] FAIL

#### Шаг f. Превышение qty / не из белого списка / флаг off -> отклонено

- Действие: три попытки. (1) qty=3 при max_contracts_per_order=2.
  (2) код не из whitelist, например SiU6. (3) quik_trading_enabled=false.
- Ожидание: каждая отклонена агентом и STL до QUIK.
- Проверка: UI — REJECTED, причина по каждой попытке (over-qty / not whitelisted /
  trading disabled). QUIK — транзакций нет. /api — три OrderUpdate
  state=ORDER_STATE_REJECTED с разными text. Команды place в логе моста нет.
- [ ] PASS  [ ] FAIL

### Этап 1b. Maker-исполнение (только после PASS всех шагов 1a)

#### Шаг g. StartExecution на N контрактов -> агент встаёт у касания, не пересекает

  Подтвердить.

  не уходит на чужую. /api — ExecutionUpdate state=working, filled растёт от 0.
  UI — прогресс исполнения, текущий рабочий объём <= max_working_contracts.

#### Шаг h. Касание сдвинулось -> наблюдаем re-quote

- Действие: дождаться сдвига нашей стороны касания на рынке (>= 1 шаг цены).
- Ожидание: агент снимает старый лимит и ставит новый у нового касания.
  Не чаще раза в 200 мс (анти-фликер).
- Проверка: QUIK — старый order_num снят, появился новый по новой цене.
  /api — серия OrderUpdate CANCELLED затем ACTIVE; ExecutionUpdate state=working.
  UI — цена рабочего лимита следует за касанием. Частота re-quote в пределах анти-фликера.

- Ожидание: исполнение останавливается. Остаток снимается. Агент не гонится,
  не становится taker (allow_cross=false).
- Проверка: QUIK — рабочих заявок не осталось. /api — ExecutionUpdate state=collar_hit,

- Проверка: QUIK — несколько сделок по нашей цене. /api — ExecutionUpdate filled

- Проверка: QUIK — все наши заявки сняты. /api — ExecutionUpdate state=stopped;

## 4. Проверка комиссии (maker)

- Все исполнения в Slice 1 идут в чистом maker-режиме. allow_cross=false.
- Заявка всегда стоит у своей стороны касания и не пересекает спред.
  Поэтому исполнение пассивное, ставка maker.
- Сверка: взять цену и объём сделки из шага c и шага j. Посчитать комиссию
  по maker-формуле группы RI из reference_commission_model. Сверить до копейки
  с комиссией в UI и в отчёте QUIK по сделке.
- Taker-сделка в чистом maker-режиме НЕДОПУСТИМА. Если сделка прошла по taker-ставке,
  это фейл: значит агент пересёк касание. Проверить, что лимит не уходил на чужую сторону.

## 5. Откат и аварийная остановка

Быстро и в порядке убывания радиуса действия.

- Kill-switch в UI. KillSwitch снимает все рабочие заявки и блокирует новые.
  Первое средство при любой нештатной ситуации.
- quik_trading_enabled = false. Мастер-флаг. Любая новая заявка отклоняется
  в агенте и в STL. Рабочие заявки снять отдельно kill-switch.
- Остановить Lua-скрипт в QUIK. Разрывает мост, новые транзакции в терминал не уходят.
  Уже стоящие в стакане заявки снять руками в QUIK.
- Крайняя мера: снять заявки вручную в окне "Заявки" QUIK и остановить агента.
- После остановки: сверить, что в QUIK не осталось рабочих заявок и открытых позиций
  сверх ожидаемых. Зафиксировать состояние.
